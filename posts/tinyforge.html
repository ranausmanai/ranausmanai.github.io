The Idea

What if a model could learn from its own wrong answers?

Not from a dataset someone else built. Not from a bigger model showing it the right way. Just from the gap between what it tried and what actually worked.

This is how humans learn to code. You write something, run it, see the error, understand why, fix it. The fix is more valuable than any tutorial — because it's specific to exactly the mistake you made.

TinyForge applies the same idea to a language model. The model tries to solve a coding problem. It fails. It sees exactly what failed — which test, what input, what it returned, what was expected. It tries again. When the second attempt is better, we extract that weak→strong pair and train on it. The model literally teaches itself from its own failures.

Self-play works for Go (AlphaGo). It works for math (AlphaProof). Here it works for code — on hardware anyone can afford.

The key is step three. Most systems just score a solution as "right" or "wrong." TinyForge captures the specific failure: input=[15] expected='FizzBuzz' got='Fizz'. That evidence is far more useful than a score. The model knows exactly what to fix.

What the model learns

Not answers. Repair patterns.

This is the most important insight. The model isn't trained on the correct solution. It's trained on the transformation from broken to working.

✗ Weak solution — 3/6 tests
def fizzbuzz(n): if n % 3 == 0: return "Fizz" if n % 5 == 0: return "Buzz" return str(n)
Fails: n=15 → "Fizz" (expected "FizzBuzz")
✓ Strong solution — 6/6 tests
def fizzbuzz(n): if n % 15 == 0: return "FizzBuzz" if n % 3 == 0: return "Fizz" if n % 5 == 0: return "Buzz" return str(n)
All 6 tests pass

The training prompt is: here's the broken code, here's the test failure, fix it. The training target is the working code. The model learns: when I see this kind of failure, I should fix it like this. It learns how to debug, not just what the answer is.

That's why it generalises. Memorising answers doesn't transfer to new problems. Learning repair patterns does.

What We Discovered

Six things that surprised us.

Finding 01

Repair beats final-answer training

Training on "here's the broken code, fix it" works better than "here's the problem, here's the solution." Small models learn fixing patterns, not memorised answers — and patterns transfer.

Finding 02

Evolutionary search is the biggest jump

Generating multiple attempts, testing them, keeping the best, mutating using failure feedback — this single mechanism accounts for more of the gain than fine-tuning alone.

Finding 03

Specific failure feedback matters a lot

expected='FizzBuzz' got='Fizz' helps more than 3/6 tests passed. Evidence beats scores every time.

Finding 04

The model learns to be a repair partner

The biggest overnight finding: the trained model writes much better code inside the feedback loop. It didn't just learn solutions — it learned how to respond to failure.

Finding 05

More compute ≠ better results

Wider search, colder sampling, extra generalisation layers — we tried them. They didn't reliably help. The simple loop outperformed every elaboration.

Finding 06

Hard problems benefit most

On easy problems where the base model already does well, the gains are small. On the hardest unseen problems — where it scored zero — repair training plus search produced real, measurable improvement.

The Bigger Picture

Code was just the test case.

The loop is simple: generate something → check if it's right → learn from what went wrong → try again. That works anywhere you can measure the output.

🗃️

SQL queries

Run the query, check if results match expectations.

test = query output
🔢

Math proofs

Verify the proof mechanically with Lean or SymPy.

test = proof verifier
🧪

Drug molecules

Check binding properties via physics simulation.

test = simulation score
📣

Ad copy

A/B test it. Did people click? Click rate is automatic feedback.

test = click rate
📧

Email outreach

Did they open? Did they reply? Open rate = the verifier.

test = reply rate
🤖

Robotics

Run the policy in simulation. The physics engine tells you if it worked.

test = simulation score

If you can score the output — even roughly — the model can learn from its own mistakes. No labelled dataset. No expert in the loop. The model makes its own training data by failing and fixing. Your data never leaves your machine.

The Real Question

If a 0.8B model can do this on a laptop — what happens at scale?

The technique is the contribution, not the model size. A 0.8B model teaching itself to code better is interesting. A 70B model with a real cluster doing the same thing might be something else entirely.

The hardware barrier to this kind of research just dropped to zero. A MacBook Air and an afternoon is enough to start.
